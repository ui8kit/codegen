Проанализировал `wp-fasty-ym`, `.workspaces/frontend` и новый движок `@ui8kit/codegen`. Ниже — слабые места, почему они ломаются при масштабировании, и куда целиться при рефакторинге.

---

## Диагноз: два разных мира

| Слой | Старый стек | Новый движок |
|------|-------------|--------------|
| **SSOT компонентов** | React + Babel-транспиляция / `data-slot` extraction | `bricks/*.def.ts` + `*.variants.json` |
| **Latte-примитивы** | Ручной HTML + `class="button button-primary"` | `generated/ui/button/Button.latte` + `UI8Kit\Classes` |
| **Гарантия DOM** | Нет | Canonical renderer + parity tests |
| **Связь WP ↔ UI** | CSS копируется в тему, шаблоны живут отдельно | Нет интеграции вообще |

Главная проблема: **frontend workspace и wp-fasty-ym не подключены к codegen**. Параллельные пайплайны дублируют работу и расходятся.

---

## 1. Сломанный пайплайн React → Latte

**Где:** `.workspaces/frontend/scripts/transforms/latte/`, `templates-builder.ts`, `react-jsdom-html.ts`

**Симптомы в продакшене** — `front-page.latte` и `navbar.latte`:

```15:47:d:\UI8Kit\Codegen\.project\wp-fasty-ym\views\front-page\front-page.latte
<section class="hero">
    ...
</section>

<section className="hero">
  <div className="container">
```

- Дублированный hero (ручной + сгенерированный)
- `className` вместо `class` — невалидный HTML/Latte
- React-атрибуты `key=`, `dateTime=` в Latte
- Смешение Tailwind (`flex items-center`) и semantic (`button button-primary`)

**Почему слабо:**
- Babel AST → Latte — хрупкий транспилятор без IR и без тестов
- Не покрывает CVA/variants, ARIA, boolean attrs, dynamic tags
- Нет обратной связи: сломался конвертер — узнаёшь только в WP

**Замена:** не конвертировать React в Latte. Использовать `bun run generate` → `generated/ui/**/*.latte` и композицию через `{capture}` + `{include}`, как в `examples/latte/templates/welcome.latte`.

---

## 2. Параллельные UI-киты без единого контракта

**Где:** `src/uikits/@ui8kit`, `@semantic`, `n3shadcn`, `n4shadcn`

Четыре копии Button/Card/Nav с разными CVA, разными `data-slot`, разными semantic-классами. README обещает «automated extraction», но:

- В `package.json` нет скриптов `semantic`, `extract-css` из README
- `TemplatesBuildConfig` указывает `uikitDir` и `templatesDir` оба на `./src/templates` — конвертер ищет не там
- `views/semantic/php/latte/` и `views/tailwind/php/twig/` — пустые `.gitkeep`

**Почему слабо:** каждый новый runtime = новый форк + новый парсер. Это ровно то, от чего codegen ушёл.

**Замена:** один registry `Codegen/bricks/` → 6 рантаймов. Frontend workspace становится **preview/authoring** для темы, а не вторым codegen.

---

## 3. Semantic CSS pipeline оторван от DOM-контракта

**Где:** `data-slot-parser.ts`, `cn-css-merge.ts`, `ui8px/*.css`, `tailwind` → `wp-fasty-ym/assets/css/theme.min.css`

Цепочка: React `cn(cva(...))` → extract `data-slot` → semantic CSS → скопировать в тему.

**Почему слабо:**
- Классы в шаблонах (`button button-primary`) не связаны с `button.variants.json`
- Смена variant в codegen не обновляет тему автоматически
- Tailwind purge/source сканирует `@ui8kit/src`, а тема использует другие классы
- Нет parity: semantic CSS может не совпасть с тем, что отрендерит Latte

**Замена:** Tailwind v4 `@source` на `generated/ui/**` + theme views (как в `examples/web/static/css/input.css`). Варианты — из `*.variants.json`, классы — через `UI8Kit\Classes::buttonClasses()` в Latte.

---

## 4. WordPress-тема: ручная вёрстка вместо примитивов

**Где:** `views/layout/parts/header.latte`, `front-page.latte`

Навигация, кнопки, карточки — сырой HTML + semantic/Tailwind классы. **Ни одного `{include}` из `generated/ui/`.**

При этом codegen уже даёт:

```17:27:d:\UI8Kit\Codegen\generated\ui\button\Button.latte
<button n:attr="[
  'type' => \UI8Kit\Rt::oneOfOr($type, 'button', 'submit', 'reset'),
  'class' => \UI8Kit\Rt::orNull(\UI8Kit\Classes::buttonClasses([...])),
  ...
]">{$children|noescape}</button>
```

**Почему слабо:** дублирование контракта кнопки/бейджа/карточки вручную; ARIA и variants расходятся с React/Templ/Vue.

**Замена:** layout/composites собирать из generated bricks:

```latte
{capture $navItems}
  {foreach $menu['primary']['items'] as $item}
    {include '../../../../generated/ui/link/Link.latte', href: $item['url'], children: $item['title']}
  {/foreach}
{/capture}
{include '../../../../generated/ui/stack/Stack.latte', class: 'gap-4', children: (string) $navItems}
```

Для `breadcrumb`, `select`, `icon` — см. [docs/php-complex-parts.md](docs/php-complex-parts.md): ручной loop + `Classes::*()`.

## 5. Layout не интегрирован с WordPress

**Где:** `views/layout/default.latte`

```24:41:d:\UI8Kit\Codegen\.project\wp-fasty-ym\views\layout\default.latte
<body class="theme-light">
    ...
    {do_action 'footer'}
</body>
```

`WordPressExtension` умеет `{wp_head}`, `{wp_footer}`, `{body_class}`, но layout их **не вызывает**. Жёсткий `theme-light`, нет `wp_enqueue` hooks в head.

**Почему слабо:** плагины (SEO, analytics, block editor styles) не подключаются; `title-tag` support в `ThemeHooks` конфликтует с ручным `<title>`.

**Замена:**

```latte
<head>
    ...
    {wp_head}
</head>
<body n:class="body_class()">
    ...
    {wp_footer}
</body>
```

## 6. Двойная обёртка и шум в инфраструктуре

| Проблема | Где |
|----------|-----|
| `<header>` в `default.latte` + `<header class="site-header">` в `header.latte` | layout |
| `listTemplates()` рекурсивно в `error_log` при каждом создании `LatteEngine` | `LatteEngine.php` |
| `error_log('ThemeService initialized')` на каждый запрос | `ThemeService.php` |
| `ContextFactory` выгружает **все** post meta в контекст | `extractPostData()` |
| Fallback autoloader + Composer autoload | `functions.php` |

**Замена:** один layout shell; debug только при `WP_DEBUG`; context — typed DTO + whitelist meta; vendor обязателен в prod.

## 7. Context: schema есть, типизации в шаблонах нет

**Где:** `context.schema.json`, `ContextFactory.php`, шаблоны с `$site['title']`

Schema на 600+ строк, но в Latte — array access без проверок. `MenuData` class_exists check, но класса нет.

**Почему слабо:** IDE не помогает; опечатка `$site['titel']` — silent fail; schema и PHP расходятся.

**Замена (поэтапно):**
1. Оставить schema как контракт **данных WP** (не UI)
2. Генерировать `context.types.php` из schema (скрипт уже есть: `schema-to-typescript.ts` → аналог для PHP)
3. UI-контекст не дублировать — примитивы получают props напрямую в `{include}`

---

## 8. Frontend workspace: runtime preview без codegen

**Где:** `app.tsx` — Elysia + `renderToStaticMarkup`, theme switcher через `require(@theme/...)`

Динамический `require` в `ComponentLoader` — хрупко для bundler/SSR. Тема переключается между `@semantic` и `@ui8kit`, но WP-тема использует третий стиль (semantic CSS files).

**Замена:** preview как `examples/latte` в Codegen:
- `generated/ui` + mock WP context JSON
- PHP built-in server или Latte render без WordPress для вёрстки
- Parity WP context fixtures ↔ canonical renderer (опционально)

## Целевая архитектура (рефакторинг)

```
Codegen/bricks/          ← SSOT ui/ примитивов (уже есть)
        ↓ generate
generated/ui/*.latte     ← примитивы для темы
generated/php/UI8Kit/    ← Rt, Classes, TwigExtension
        ↓ compose
wp-fasty-ym/views/       ← только layout + WP composites (PostCard, ArchiveList)
        ↓ context
ContextFactory           ← только WP data (site, menu, posts)
        ↓ render
LatteEngine + WPExtension
```

### Что удалить / заморозить

| Удалить | Причина |
|---------|---------|
| `scripts/transforms/latte/` (JSX→Latte) | Заменён codegen emitter |
| `react-jsdom-html.ts` semantic substitution | Нет DOM-гарантии |
| `data-slot-parser` → `@semantic` fork | Дублирует variants.json |
| `n3shadcn` / `n4shadcn` копии | Один registry |
| Дубли hero в `front-page.latte` | Мусор от конвертера |
| `front-page-old.latte` | Legacy |

### Что оставить и усилить

| Оставить | Доработать |
|----------|------------|
| `WordPressExtension` | + helper `{ui include 'button/Button', ...}` |
| `ContextFactory` | Slim context, typed, без all-meta dump |
| `context.schema.json` | Генерация PHP types |
| DI Container | Убрать debug noise |
| `ThemeService` two-phase render | layout + content — ок, но content = composites из bricks |

### Новые артефакты (предложение)

```
.project/
  theme-builder/              # или расширить Codegen
    bricks-wp/                # WP composites: PostCard, SiteHeader (def.ts)
    fixtures/wp-context.json  # parity fixtures для theme views
  wp-fasty-ym/
    composer.json             # + path repo на generated/php
    views/                    # только layout + composites
    vendor/ui8kit/            # symlink/copy generated/ui + php/
```

**Build pipeline темы:**

```bash
cd Codegen && bun run generate --runtimes latte
rsync generated/ui wp-fasty-ym/vendor/ui8kit/ui
rsync generated/php wp-fasty-ym/vendor/ui8kit/php
cd .workspaces/frontend && bun run tailwind   # @source generated + views
```

## Итог

Старый подход строился на идее **«React prototype → transpile → semantic CSS → WP»** — без executable spec и с четырьмя форками UI-кита.

Новый движок даёт **«brick def → N runtimes → parity tests»**. Для WordPress+Latte правильная модель:

- **Примитивы** — только из `generated/ui/*.latte`
- **Тема** — layout + WP context + композиция `{capture}/{include}`
- **Сложные parts** (breadcrumb, select, icon) — loops в theme views + `Classes::*()`
- **Frontend workspace** — preview/build CSS, не второй codegen

Могу следующим шагом: (a) набросать `docs/wp-theme-migration.md` в репозитории, (b) сделать POC — `header.latte` на generated `Button`/`Link`/`Stack`, (c) wiring `composer.json` темы на `generated/`. Что предпочтительнее?
